## 7.1 Experimental Environment

All experiments reported in this study were executed within a single, version-controlled software environment designed to support reproducible training, evaluation, and post-hoc analysis of the PF-STGT framework described in Chapter 6. The environment couples a pinned Python scientific stack with PyTorch-based deep learning modules, classical machine-learning routines, and explainability tooling. Hardware utilisation follows a portable device-selection policy so that the same codebase can run on CUDA-capable GPUs, Apple Metal accelerators, or CPU fallback without altering model definitions or data pipelines. This subsection documents the compute platform, software dependencies, and reproducibility controls that bound the experimental work; training hyperparameters and evaluation protocols are specified in subsequent subsections.

**Compute platform.** Deep-learning experiments select the execution device automatically at runtime using the priority order CUDA, Metal Performance Shaders (MPS), then CPU. When an NVIDIA GPU with CUDA support is available, training and inference use it; otherwise, the PyTorch MPS backend is selected if present; if neither accelerator is available, computation proceeds on CPU. This policy is implemented uniformly across training, ablation, benchmark, and explainability scripts. Frozen reference runs for the proposed PF-STGT models—including initial training, ablation studies, and explainability analysis—were executed on the **MPS** backend, as recorded in experiment metadata. Classical benchmark models (linear regression, random forest, and gradient boosting) are trained and evaluated on **CPU** without loading the PyTorch deep-learning stack, isolating lightweight tabular learners from accelerator-dependent code paths. Reproducibility is anchored on the documented compute backend and pinned software versions used for the frozen experimental programme.

**Software environment.** Experiments were conducted in a Conda environment named `smartgrid-stgt` with **Python 3.11**, as specified in `environment.yml`. Python package versions are additionally pinned in `requirements.txt` to lock the dependency graph of the publication-freeze version of the repository (25 June 2026). The deep-learning core comprises **PyTorch 2.3.1** \cite{Paszke2019PyTorch} and **torchvision 0.18.1**. Graph-aware model components rely on **PyTorch Geometric 2.5.3** \cite{Fey2019PyG} and **NetworkX 3.3**. The numerical and tabular data stack includes **NumPy 1.26.4**, **Pandas 2.2.2**, **SciPy 1.13.1**, and **scikit-learn 1.5.0** \cite{Pedregosa2011ScikitLearn}. Post-hoc explainability modules draw on **SHAP 0.45.1** \cite{Lundberg2017SHAP} and **Captum 0.7.0** \cite{Kokhlikyan2020Captum}. Configuration management, logging, and experiment orchestration use **PyYAML 6.0.1**, **Hydra 1.3.2**, **OmegaConf 2.3.0**, and **tqdm 4.66.4**. Visualisation and notebook utilities (**Matplotlib 3.9.0**, **Seaborn 0.13.2**, **JupyterLab 4.2.2**) support diagnostic plotting but are not on the critical path for frozen result generation. **CUDA** is supported by the device-selection policy when a compatible NVIDIA driver and GPU are present; the frozen reference experiments documented in the publication package executed on the **MPS** backend, and no CUDA toolkit version is recorded for those runs.

**Reproducibility controls.** Three layers of environmental control complement the methodological safeguards in Sections 6.3, 6.7, and 6.8. First, dependency versions are frozen in `requirements.txt` and `environment.yml`, enabling reconstruction of the interpreter and library stack. Second, a fixed random seed (42) governs weight initialisation, data-loader shuffling, and stochastic operations in the reference deep-learning runs, with seed-setting utilities applied consistently across NumPy, Python, and PyTorch. Third, cleaned datasets, engineered feature representations, and graph specifications are verified for consistency against the frozen experimental inputs established in the methodology, ensuring that each run consumes identical data and graph artefacts. All reported results were generated from the publication-freeze version of the repository and are not regenerated during manuscript preparation. Together, the pinned software environment, portable compute policy, and input consistency checks provide a reproducible execution context appropriate for a daily-resolution national grid dataset whose training partition contains fewer than 1,300 sliding-window samples.

**Environmental suitability.** The chosen environment matches the scale and structure of the study. PyTorch and PyTorch Geometric supply native support for batched graph-tensor operations, transformer blocks, and automatic differentiation required by PF-STGT. The automatic CUDA/MPS/CPU fallback allows development and frozen experimentation on Apple Metal hardware while preserving compatibility with CUDA workstations used in typical deep-learning workflows. Pinned versions of NumPy, Pandas, and scikit-learn ensure that classical baselines and evaluation statistics are computed with the same numerical libraries as the deep-learning pipeline. CPU execution for classical models avoids unnecessary accelerator overhead on low-parameter learners. The overall configuration prioritises reproducibility and moderate compute requirements over distributed or large-cluster training, which is proportionate to the dataset size and model capacity established in Chapter 6.

## 7.2 Baseline Models

Benchmark comparisons evaluate the proposed forecasting framework against seven reference models (B01–B07) \cite{Breiman2001RandomForest,Chen2016XGBoost,Hochreiter1997LSTM,Cho2014GRU,Zhao2020TGCN} spanning classical machine learning, recurrent deep learning, and graph-based spatio-temporal learning. The suite forms a progressive complexity ladder so that performance differences can be attributed to identifiable methodological families. Baselines B01–B06 target Task 1 (regional demand) as the primary comparison axis; B07 alone supports joint Task 1 and Task 2 (OSI) prediction as the historical PF-STGT internal reference. The final proposed model, S2 (correlation-graph PF-STGT), is evaluated separately in the ablation study and is not assigned a B-series identifier.

**Classical machine-learning baselines (B01–B03).** Three tabular learners provide low-to-moderate complexity reference points that are widely used in load-forecasting literature and require no graph or sequence-specialised architecture.

*B01 — Linear Regression* represents the simplest multivariate statistical baseline. Inputs consist of the seven-day engineered feature window flattened into a single vector per forecast origin, combining regional node features and national global features. Demand is predicted with an ordinary least-squares multi-output regressor across the nine divisions. Linear regression establishes a transparent lower bound on forecast accuracy and tests whether the engineered feature set alone carries predictive signal without non-linear interactions.

*B02 — Random Forest* \cite{Breiman2001RandomForest} belongs to the ensemble tree family and captures non-linear feature interactions and variable importance through bagged decision trees. Random forest is included because ensemble methods form a dominant cluster in hybrid load-forecasting studies and frequently achieve strong performance on tabular engineered features without explicit temporal or spatial structure. The model is trained under its frozen optimised configuration (Section 7.4).

*B03 — XGBoost* \cite{Chen2016XGBoost} implements gradient-boosted decision trees with histogram-based training. XGBoost complements random forest within the classical tier by representing the boosting family, which often excels on heterogeneous tabular inputs. Together, B01–B03 isolate the value of non-linearity and ensemble capacity relative to the deep-learning and graph-based tiers. Complete hyperparameter settings are reported in Section 7.4.

For all classical models, node and global tensors are standardised using a scaler fit exclusively on the training partition before regression. Although the primary benchmark comparison focuses on demand forecasting, each classical model also fits a separate scalar regressor for OSI so that stress metrics can be reported on a common evaluation footing; these stress predictions are secondary and do not define the classical tier’s methodological role.

**Deep temporal baselines (B04–B05).** Recurrent neural networks test whether sequence modelling without explicit graph structure can capture the temporal dependencies encoded in the seven-day input window.

*B04 — LSTM* \cite{Hochreiter1997LSTM} employs a two-layer long short-term memory network with hidden dimension 128 and inter-layer dropout of 0.1. At each time step, regional node features and global features are concatenated into a single input vector; shared recurrent weights process the sequence and a linear head outputs nine regional demand forecasts at the one-day horizon. LSTM is selected because recurrent architectures remain a standard deep-learning baseline in short-horizon energy forecasting and isolate temporal recurrence from graph-based spatial coupling.

*B05 — GRU* \cite{Cho2014GRU} follows the identical input encoding and head structure as B04 but replaces LSTM cells with gated recurrent units. Including both LSTM and GRU controls for cell-type variation within the recurrent family and tests whether gating complexity materially affects demand forecasts on daily windows of length \(T = 7\).

Both models are trained under the demand-primary protocol applied to B01–B06. An auxiliary stress head enables comparable OSI evaluation but is not the primary training objective.

**Graph-based spatio-temporal baseline (B06).** *B06 — T-GCN* \cite{Zhao2020TGCN} is the sole graph neural network in the benchmark suite external to PF-STGT. At each time step, regional node features are projected, aggregated through single-hop graph convolution with the row-normalised hybrid adjacency matrix from Section 6.5, and combined with projected global features before temporal aggregation by a GRU. T-GCN isolates graph convolution plus recurrent pooling without the parallel transformer fusion or multi-task heads of PF-STGT. B06 and B07 share the same hybrid adjacency prior for a controlled comparison against graph-transformer fusion.

**Multi-task PF-STGT reference (B07).** *B07 — PF-STGT (W20)* is the historical PF-STGT reference configuration used as an internal baseline within the benchmark suite. It implements the parallel-fusion spatio-temporal graph transformer described in Section 6.6 under the W20 multi-task protocol (Section 6.8), with hybrid geographical–correlation graph conditioning. B07 jointly predicts regional demand and graph-level OSI from a shared latent representation and serves as the multi-task graph-transformer reference against which B01–B06 are compared on Task 1; it alone supports primary Task 2 evaluation among the B-series models. S2, the final proposed model (correlation-only graph PF-STGT), is not a B-series benchmark identifier: it is evaluated separately in the ablation study (configuration A6) and supersedes B07 as the publication-facing architecture while retaining the same training protocol documented in Table 2.

**Fairness protocol.** All benchmark models adhere to a common experimental contract designed to prevent information leakage and to ensure that performance differences reflect modelling choices rather than data-handling inconsistencies.

*Data partitioning.* Every model uses the identical chronological train, validation, and test partitions defined in Section 6.2 and Table 1. No model accesses validation or test observations during training except through the prescribed model-selection procedures described below. The first seven rows of each partition are excluded as warm-up, and sliding windows of length \(T = 7\) with horizon \(H = 1\) are constructed identically across models.

*Preprocessing and features.* All models read from the same frozen engineered feature store produced by the preprocessing and feature-engineering pipeline in Sections 6.3 and 6.4. Leakage policies established in Section 6.1 apply uniformly: same-day OSI is excluded from inputs when predicting next-day OSI, and normalisation statistics for stress targets are fit on the training partition only. Classical models apply an additional standardisation step to flattened inputs, with scaler parameters fit on training data only.

*Feature availability and graph priors.* Models that do not use graph structure (B01–B05) consume the same node-level and global feature tensors reduced to tabular or sequential form. Graph-based models (B06, B07) additionally receive the shared hybrid adjacency matrix and, for B07, the derived attention-bias matrix. No baseline is given access to features, graph edges, or targets unavailable to the others at the same forecast origin.

*Evaluation metrics.* Task 1 performance is assessed with macro-averaged mean absolute error (MAE), root mean squared error (RMSE), mean absolute percentage error (MAPE), and coefficient of determination (R²) computed per region on the held-out test split and averaged over the nine divisions, as specified in the frozen evaluation protocol. Task 2 metrics (MAE, RMSE, R²) apply to models that produce OSI forecasts. The test split is evaluated once after model selection; no test-set tuning is permitted.

*Hyperparameter tuning.* Deep-learning baselines (B04–B06) follow a common optimisation protocol—shared optimiser, learning-rate schedule, early stopping, and batch configuration—described in Section 7.4. Classical models (B01–B03) use frozen library configurations selected under the same validation-based protocol. All hyperparameter values for the reference runs are consolidated in Section 7.4 and Table 2. Seed 42 governs the frozen reference experiments.

## 7.3 Evaluation Metrics

Model performance is assessed on the held-out test partition defined in Section 6.2 after all training and model-selection decisions have been finalised on the training and validation splits. Evaluation is task-specific: Task 1 (regional evening-peak demand) is the primary comparison axis for all benchmark models in Section 7.2, whereas Task 2 (Operational Stress Index forecasting) is reported for multi-task configurations that produce graph-level OSI outputs. This subsection specifies the metric definitions, aggregation rules, and interpretation principles that govern all quantitative reporting in subsequent results sections. A single metric is insufficient because demand and stress targets differ in scale, units, and operational meaning; the frozen protocol therefore reports a compact set of complementary error and goodness-of-fit measures for each task.

**Evaluation scope.** All metrics in this study are computed on the test split containing 264 valid sliding-window forecast origins (Table 1). Predictions \(\hat{\mathbf{y}}^{\mathrm{demand}}\) and \(\hat{y}^{\mathrm{osi}}\) are compared against the targets prepared in Sections 6.1 and 6.7 without refitting scalers, graph parameters, or model weights. Validation metrics guide training and hyperparameter selection; test metrics are evaluated once and constitute the primary record for publication.

**Task 1 — Regional demand metrics.** For each region \(r \in \{1, \ldots, N\}\) with \(N = 9\), let \(D_{r,i}\) denote the observed evening-peak demand (MW) and \(\hat{D}_{r,i}\) the model prediction on test sample \(i\), where \(i = 1, \ldots, n\) and \(n\) is the number of test windows. Per-region metrics are computed first; national summary scores are obtained by macro-averaging across regions unless stated otherwise.

*Mean absolute error (MAE)* measures average forecast deviation in megawatts, the native unit of grid demand:

\[
\mathrm{MAE}_r = \frac{1}{n} \sum_{i=1}^{n} \left| D_{r,i} - \hat{D}_{r,i} \right|, \qquad
\mathrm{MAE}_{\mathrm{macro}} = \frac{1}{N} \sum_{r=1}^{N} \mathrm{MAE}_r.
\]

MAE is the **primary ranking metric** for Task 1 because it is expressed in directly interpretable physical units, is robust to moderate outliers relative to squared-error measures, and aligns with operator-facing dispatch error reporting.

*Root mean squared error (RMSE)* penalises large regional forecast errors more heavily than MAE:

\[
\mathrm{RMSE}_r = \sqrt{\frac{1}{n} \sum_{i=1}^{n} \left( D_{r,i} - \hat{D}_{r,i} \right)^2}, \qquad
\mathrm{RMSE}_{\mathrm{macro}} = \frac{1}{N} \sum_{r=1}^{N} \mathrm{RMSE}_r.
\]

RMSE is reported alongside MAE to indicate whether a model's error distribution is dominated by occasional large misses, which is relevant for reserve and security assessment.

*Mean absolute percentage error (MAPE)* expresses relative error as a percentage, supporting comparison with load-forecasting studies that report scale-normalised accuracy:

\[
\mathrm{MAPE} = \frac{100}{|\mathcal{V}|} \sum_{(r,i) \in \mathcal{V}} \frac{\left| D_{r,i} - \hat{D}_{r,i} \right|}{\left| D_{r,i} \right|},
\]

where \(\mathcal{V}\) is the set of valid region-day pairs with non-negligible observed demand (\(|D_{r,i}| > \varepsilon\)) to avoid division instability. MAPE is reported for cross-paper comparability but is not used for model selection because it is sensitive to low-demand days and regional scale differences.

*Coefficient of determination (R²)* quantifies the proportion of regional demand variance explained by the model on the test split:

\[
R^2_r = 1 - \frac{\sum_{i=1}^{n} \left( D_{r,i} - \hat{D}_{r,i} \right)^2}{\sum_{i=1}^{n} \left( D_{r,i} - \bar{D}_r \right)^2}, \qquad
R^2_{\mathrm{macro}} = \frac{1}{N} \sum_{r=1}^{N} R^2_r,
\]

where \(\bar{D}_r\) is the mean observed demand in region \(r\) over the test period. **Macro R²** is defined as the unweighted mean of per-region \(R^2_r\) values, not as a single pooled \(R^2\) over all region-day pairs. This aggregation treats each division equally and prevents high-load regions from dominating the variance-explained statistic. R² complements MAE by indicating how well a model tracks regional demand variability; the two metrics need not rank models identically when prediction variance differs from observed variance.

In addition to macro summaries, **Dhaka** division MAE is reported separately. Dhaka carries the largest share of national evening-peak load and is the principal demand centre for national transmission and day-ahead dispatch planning; isolating its forecast error therefore provides an operationally focused diagnostic beyond the unweighted macro average. Per-region MAE values for all nine divisions are available in the frozen results package for further analysis.

**Task 2 — Operational stress metrics.** Task 2 evaluates graph-level OSI predictions \(\hat{y}^{\mathrm{osi}}_i\) against targets \(y^{\mathrm{osi}}_i \in [0, 1]\) on the same test windows. Because OSI is a bounded scalar index rather than a nine-dimensional regional vector, metrics are computed directly over the \(n\) test samples without regional macro-averaging:

\[
\mathrm{MAE}_{\mathrm{osi}} = \frac{1}{n} \sum_{i=1}^{n} \left| y^{\mathrm{osi}}_i - \hat{y}^{\mathrm{osi}}_i \right|,
\]

\[
\mathrm{RMSE}_{\mathrm{osi}} = \sqrt{\frac{1}{n} \sum_{i=1}^{n} \left( y^{\mathrm{osi}}_i - \hat{y}^{\mathrm{osi}}_i \right)^2},
\]

\[
R^2_{\mathrm{osi}} = 1 - \frac{\sum_{i=1}^{n} \left( y^{\mathrm{osi}}_i - \hat{y}^{\mathrm{osi}}_i \right)^2}{\sum_{i=1}^{n} \left( y^{\mathrm{osi}}_i - \bar{y}^{\mathrm{osi}} \right)^2}.
\]

MAE\(_{\mathrm{osi}}\) is the primary stress error measure because OSI lies on a unit interval and absolute deviation is directly interpretable as forecast bias on the normalised stress scale. RMSE\(_{\mathrm{osi}}\) highlights large stress-prediction failures that may correspond to under-predicted operational pressure. \(R^2_{\mathrm{osi}}\) indicates whether the model captures day-to-day stress variability beyond the test-set mean. **Pearson correlation** \(r(\hat{y}^{\mathrm{osi}}, y^{\mathrm{osi}})\) is reported only as a supplementary descriptive statistic for multi-task models and is not used for model ranking or selection. Simple reference forecasts—persistence (\(\hat{y}^{\mathrm{osi}}_{i} = y^{\mathrm{osi}}_{i-1}\)) and the training-set median—provide contextual baselines for interpreting learned stress predictions; they are not benchmark models defined in Section 7.2.

**Macro averaging and metric interpretation.** Macro-averaging for Task 1 assigns equal weight to each administrative division, consistent with the node-level forecasting formulation in Section 6.1 and preventing a single high-load region from determining the national leaderboard score. This choice reflects the study's objective of equitable regional forecast quality across Bangladesh's nine divisions rather than optimising exclusively for the national aggregate. For Task 2, no macro average is required because supervision is graph-level.

Multiple metrics are reported jointly because each captures a distinct facet of forecast quality: MAE and RMSE quantify magnitude of error in physical or normalised units; MAPE supports literature comparison; and R² assesses variance explanation. Relying on a single score—particularly R² alone—can obscure models that achieve low absolute error but under-represent temporal variability, or vice versa. The frozen protocol designates **macro demand MAE on the test split** as the sole primary metric for ranking Task 1 models; all other Task 1 and Task 2 measures are secondary and interpretive. Formal inferential comparison of paired forecast errors, including Wilcoxon signed-rank testing \cite{Wilcoxon1945SignedRank} and Bonferroni correction \cite{Dunn1961Bonferroni}, is specified in Section 7.5 (Statistical Testing); numerical outcomes are reported in Section 8.

## 7.4 Hyperparameter Configuration

The evaluation metrics in Section 7.3 are computed under a fixed hyperparameter regime established before the publication freeze and held constant across all reference runs. Hyperparameters were selected on the validation split only; the test partition was not used for tuning. A fixed random seed (42) governs all stochastic components in the frozen reference experiments. This subsection records the training and model-capacity settings for the proposed PF-STGT framework (S2 / ablation A6), the deep-learning baselines (B04–B06), and the classical baselines (B01–B03). **Table 2** provides the authoritative summary for the final proposed model; the narrative below supplements Table 2 with architecture-capacity parameters and baseline-specific settings that fall outside the table.

**Shared windowing parameters.** All models share the input sequence length \(T = 7\) days and one-day forecast horizon \(H = 1\), consistent with Sections 6.1 and 6.2. These values are fixed by the dataset design and are not treated as tunable hyperparameters.

**Proposed PF-STGT configuration (S2 / A6).** The final proposed model uses the W20 multi-task training protocol with correlation-only graph conditioning. Table 2 consolidates the optimisation settings: **Adam** \cite{Kingma2015Adam} optimiser with learning rate \(\eta = 5 \times 10^{-4}\) and weight decay \(\lambda_{\mathrm{wd}} = 10^{-4}\); mini-batch size \(B = 32\); maximum training duration of 200 epochs subject to early stopping with **patience 15** on the balanced validation score \(0.7 \cdot (\mathrm{MAE}_{\mathrm{demand}}^{\mathrm{val}} / 100) + 0.3 \cdot \mathrm{MAE}_{\mathrm{osi}}^{\mathrm{val}}\); **ReduceLROnPlateau** scheduling on validation macro demand MAE (multiplicative factor 0.5, scheduler patience 5); and **gradient-norm clipping** at maximum norm 1.0. Multi-task loss weighting uses stress coefficient \(\lambda_2 = 20.0\) with **demand-loss scaling** that divides the Huber demand term by 100 MW before combination with OSI mean squared error, as specified in Section 6.8.

Architectural capacity hyperparameters for PF-STGT were fixed during architecture selection: hidden dimension \(d_{\mathrm{model}} = 128\), four attention heads, two spatial transformer layers and two temporal transformer layers, feed-forward inner dimension 256, and **dropout** rate 0.1 in both spatial and temporal transformer branches. The stress-prediction head uses an internal hidden dimension of 128. These capacity choices balance representational power against the approximately 1,281 training windows available after warm-up exclusion.

**Deep-learning baseline configuration (B04–B06).** Recurrent baselines B04 (LSTM) and B05 (GRU) employ a two-layer sequence encoder with hidden dimension 128 and inter-layer dropout of 0.1; graph baseline B06 (T-GCN) uses hidden dimension 64 in its convolutional and temporal blocks. Deep baselines are trained with **AdamW** \cite{Loshchilov2019AdamW} (learning rate \(5 \times 10^{-4}\), weight decay \(10^{-4}\)), batch size 32, gradient-norm clipping at 1.0, ReduceLROnPlateau scheduling on validation macro demand MAE (factor 0.5, patience 5), and early stopping with patience 15 on validation demand MAE. Training runs for up to 60 epochs in the frozen benchmark experiment, retaining the checkpoint with lowest validation demand MAE. Demand fit uses macro-averaged Huber loss during training; these baselines follow the demand-primary protocol described in Section 7.2.

**Classical baseline configuration (B01–B03).** Classical models operate on flattened seven-day feature vectors with **standardisation** parameters fit exclusively on the training partition. B01 (linear regression) uses ordinary least-squares multi-output regression for demand without additional regularisation hyperparameters. B02 (random forest) and B03 (XGBoost) use frozen ensemble configurations selected during validation-based hyperparameter tuning: 300 decision trees for random forest, and for XGBoost 300 estimators with maximum tree depth 6 and learning rate 0.1. Classical fits are deterministic given the fixed seed and feature matrix; no iterative early stopping applies.

**Configuration suitability.** The frozen hyperparameter set is proportionate to the daily-resolution dataset and the multi-task objective. Batch size 32 and learning rate \(5 \times 10^{-4}\) were among the candidates evaluated during validation-based hyperparameter tuning and provide stable convergence across PF-STGT and deep baseline architectures. Capacity parameters (\(d_{\mathrm{model}} = 128\), two transformer layers per branch) avoid over-parameterisation relative to the training-window count while preserving the spatial and temporal representational depth required by the graph-transformer design. Multi-task settings (\(\lambda_2 = 20\), demand scaling by 100 MW) are fixed from the W20 protocol established before ablation and benchmark reporting. Gradient clipping and weight decay supply mild regularisation without altering the loss definitions in Section 6.8. Classical ensemble configurations were chosen to represent strong tabular competitors at moderate computational cost on CPU. Together, Table 2 and the settings above define the complete reproducible hyperparameter record for all experiments reported in this study.

## 7.5 Statistical Testing

The hyperparameter configuration in Section 7.4 fixes the models under comparison; this subsection specifies how paired forecast errors on the held-out test split are tested for systematic differences. Inferential analysis applies exclusively to **Task 1** and uses the primary evaluation metric defined in Section 7.3—macro-averaged demand MAE in megawatts—computed independently for each test-window forecast origin. Comparisons are organised into three pre-specified families: benchmark models (reference PF-STGT versus B01–B06), ablation variants (reference A1 versus A2–A6), and architecture-simplification variants (reference S1 versus S2–S4). Numerical test outcomes are reported in Section 8; the present subsection defines the protocol only.

**Paired comparison unit.** For each test window \(i = 1, \ldots, n\) with \(n = 264\), every model produces a vector of nine regional demand forecasts. The paired statistic is per-window macro MAE:

\[
\mathrm{MAE}_i = \frac{1}{N} \sum_{r=1}^{N} \left| D_{r,i} - \hat{D}_{r,i} \right|,
\]

with \(N = 9\). Because all models are evaluated on the identical set of forecast origins, differences \(\Delta\mathrm{MAE}_i = \mathrm{MAE}_i^{(\mathrm{ref})} - \mathrm{MAE}_i^{(\mathrm{cand})}\) form paired observations across the same test days. This pairing controls for day-specific grid conditions—calendar effects, weather, and operational events—that would confound unpaired comparisons.

**Wilcoxon signed-rank test.** The primary inferential test is the **paired Wilcoxon signed-rank test** \cite{Wilcoxon1945SignedRank} applied to \(\{\Delta\mathrm{MAE}_i\}\). The null hypothesis is that the median paired difference is zero. A non-parametric test is appropriate because per-window forecast errors are typically skewed, may contain occasional large deviations on high-load or stressed days, and are not assumed to follow a Gaussian distribution. The signed-rank procedure uses the magnitudes of non-zero differences while respecting their signs, making it robust to outliers relative to paired \(t\)-tests. **Two-sided** \(p\)-values are reported as the default for general model comparisons. **One-sided** \(p\)-values are computed only where a pre-registered directional hypothesis specifies that the reference model should improve upon a particular baseline; Bonferroni-adjusted significance for such claims uses the appropriate one-sided probability.

**Multiple-comparison correction.** Each comparison family involves several simultaneous hypothesis tests. **Bonferroni correction** \cite{Dunn1961Bonferroni} controls the family-wise error rate by adjusting the nominal significance level. For the benchmark family (six pairwise comparisons of the reference PF-STGT model against B01–B06), the adjusted threshold is \(\alpha_B = 0.05 / 6 \approx 0.0083\). For the ablation family (five comparisons of A1 against A2–A6), \(\alpha_B = 0.05 / 5 = 0.01\). The architecture-simplification family applies the same \(\alpha_B = 0.01\) criterion for comparisons against S1. A comparison is declared statistically significant only when the relevant one-sided or two-sided \(p\)-value falls below the family-specific Bonferroni-adjusted threshold and the direction of the median paired difference is consistent with the stated hypothesis.

**Effect size and uncertainty quantification.** Statistical significance is supplemented by **effect size** and **bootstrap confidence intervals** to support interpretation. For benchmark comparisons, **Cohen's \(d\)** \cite{Cohen1988StatisticalPower} is computed from the paired differences as the mean \(\Delta\mathrm{MAE}\) divided by the sample standard deviation of \(\Delta\mathrm{MAE}\), with \(|d| \geq 0.5\) treated as a medium effect in reporting guidelines. Cohen's \(d\) is reported as a standardised descriptive measure of practical effect magnitude that complements the non-parametric Wilcoxon test; it does not replace a rank-based effect-size statistic but aids interpretation of the megawatt-scale paired differences on a common scale. **Bootstrap 95% confidence intervals** \cite{Efron1993Bootstrap} on the mean paired MAE difference are obtained by resampling test windows with replacement (2,000 resamples, fixed random seed) and taking the 2.5th and 97.5th percentiles of the resampled mean differences. Bootstrap intervals provide distribution-free uncertainty estimates for the mean paired improvement or degradation without assuming normality of the error differences.

**Statistical significance versus practical importance.** Rejecting the null hypothesis indicates that a model's per-window macro MAE differs systematically from its paired reference on the test split; it does not, by itself, establish operational value. **Practical importance** is assessed from the magnitude of paired MAE differences expressed in megawatts, from bootstrap confidence intervals that exclude zero, and from effect-size conventions. A statistically significant improvement of very small megawatt magnitude may be inconsequential for national dispatch planning, whereas a moderate MAE reduction with a medium Cohen's \(d\) and a bootstrap interval entirely below zero provides stronger evidence of meaningful gain. Conversely, a non-significant test does not prove equivalence; insufficient test-window count or high day-to-day variance may limit power to detect modest but operationally relevant differences. Section 8 presents these outputs in the order encountered by the reader: raw MAE metrics first, followed by bootstrap confidence intervals, then \(p\)-values with Bonferroni-adjusted thresholds, and finally Cohen's \(d\) where applicable—with macro demand MAE remaining the primary ranking criterion from Section 7.3.
